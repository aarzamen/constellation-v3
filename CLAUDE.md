# CLAUDE.md — Agent Orientation for Constellation V3

## Project Overview

Constellation is a **self-hosted semantic memory server** for AI conversation history exports (Claude, ChatGPT, Gemini, Grok). It provides:
1. A local embedding + clustering pipeline (Python/numpy)
2. An MCP-compatible memory server for LLM clients (via `fastmcp` over stdio)
3. A REST API for search/retrieval
4. An interactive 3D visualization frontend (`3d-force-graph` + Three.js)

**Product intent**: The MCP memory server is the primary interface — the 3D visualization is a development/exploration tool, not the product surface. Design decisions should prioritize the MCP tool-use experience for LLM clients (both chat agents and coding agents).

**Single developer project, MIT licensed, no CI/CD pipeline.**

## Multi-Provider Support

Constellation indexes conversations from multiple AI providers in a unified
semantic space. Conversations cluster by **topic**, not by provider.

### Supported Providers

| Provider | Parser | Format | Visual Shape |
|---|---|---|---|
| **Claude** (Anthropic) | `core/parser.py` | Flat message array, ISO timestamps | Sphere |
| **ChatGPT** (OpenAI) | `core/chatgpt_parser.py` | DAG/tree with `mapping` field, Unix epoch | Octahedron |
| **Gemini** (Google) | `core/gemini_parser.py` | Folder of extensionless JSON files | Dodecahedron |
| **Grok** (xAI) | `core/grok_parser.py` | Stub — format auto-detection | Icosahedron |

### Ingesting Additional Providers

```bash
# Add ChatGPT data
python launch.py --chatgpt-source /path/to/chatgpt/conversations.json

# Generic multi-provider flag
python launch.py --add-source chatgpt /path/to/chatgpt/conversations.json
python launch.py --add-source gemini /path/to/gemini/folder
```

After first run, source paths are saved in `config.yaml` under `sources:` and
automatically included in subsequent runs without requiring CLI flags.

### ChatGPT Export Notes

- File uses a tree structure (`mapping` field with parent/children references)
- Parser linearizes using `current_node` backward traversal — only the active branch is extracted
- System messages, tool outputs, and DALL-E generations are skipped
- Deleted messages (weight=0) are excluded

### Gemini AI Studio Notes

- Each conversation is a separate extensionless JSON file in a Google Drive folder
- Parser scans directory, detects valid chat files by checking for `chunkedPrompt` marker
- Thinking blocks (`isThought: true`) are excluded
- No per-message timestamps (only conversation-level)

## Key Design Constraints

- **No external vector DBs** — all math is numpy-based (kmeans, PCA, cosine sim)
- **No frameworks** — no LangChain, LlamaIndex, Flask, FastAPI, React, etc.
- **No build step** — frontend is vanilla JS served from filesystem
- **MCP over stdout** — `mcp_server.py` uses stdio; ALL print statements MUST go to `sys.stderr` to avoid corrupting the JSON-RPC protocol
- **Chunk-level embeddings** — user messages are chunked (~150 words) and individually embedded; search operates on chunks then maps back to conversations

## MCP Memory Server Interface

The MCP server (`server/mcp_server.py`) exposes Constellation as a tool-use memory layer for LLM clients. It runs as a **standalone subprocess** over stdio (JSON-RPC), designed to be launched by Claude Desktop or any MCP-compatible host.

### MCP Tools Exposed

| Tool | Args | Returns | Write? |
|---|---|---|---|
| `search_conversations` | `query: str`, `top_k: int = 5` | List of matches with title, date, scores, excerpt, cluster_id, notes_count | No |
| `get_conversation` | `conversation_id: str` (UUID) | Full conversation with messages, notes, cluster_id | No |
| `list_conversations` | `offset: int = 0`, `limit: int = 20`, `sort_by: str = "date"` | Paginated conversation list (max 50) | No |
| `get_stats` | *(none)* | Index statistics (counts, date range, embedding info) | No |
| `add_conversation_note` | `conversation_id: str`, `note_text: str` | Status dict with new note and current notes list | Yes (sidecar `data/notes.json`) |
| `delete_conversation_note` | `conversation_id: str`, `note_id: str` | Status dict with remaining notes list | Yes (sidecar `data/notes.json`) |

### Search Pipeline (what happens inside `search_conversations`)

1. Query is embedded via `all-MiniLM-L6-v2` (384d vector)
2. Three scoring passes run in parallel:
   - **Conversation-level semantic**: cosine similarity of query vs. mean-pooled conversation embeddings
   - **Chunk-level semantic**: cosine similarity of query vs. individual user message embeddings (finds the single best-matching message per conversation)
   - **Lexical (BM25)**: token-level keyword matching across conversation text
3. Scores are fused via **Reciprocal Rank Fusion (RRF)** with k=60
4. Top-k results returned with the best-matching message excerpt auto-extracted

### Result Shape (per search hit)

```json
{
  "id": "uuid-string",
  "title": "Conversation Name",
  "date": "2025-03-08T00:00:00.852470Z",
  "score": 0.032,
  "conversation_score": 0.71,
  "chunk_score": 0.84,
  "lexical_score": 3.2,
  "message_count": 12,
  "excerpt": "The exact user message that matched best...",
  "cluster_id": 3,
  "cluster_label": "deployment / medical / readiness",
  "notes_count": 1
}
```

### Notes System (`add_conversation_note` / `delete_conversation_note`)

- Notes are **append-only metadata** — they do not alter embeddings, clusters, or graph data
- Notes are persisted in a **sidecar file** (`data/notes.json`), separate from `conversations.json`
- Each note stores `{text, created_at, note_id}` with a UTC ISO timestamp and 8-char hex ID
- Notes **survive pipeline rebuilds** including `--reembed` — the sidecar is never overwritten by the pipeline
- Atomic writes via `os.replace` prevent partial writes
- Legacy notes in `conversations.json` are automatically migrated to the sidecar on first load
- Use `delete_conversation_note` with the `note_id` to remove a note (no update — delete and re-add)

### REST API Equivalents (HTTP Server)

The HTTP server at port 8420 exposes the same `SearchEngine` class via REST:

| Endpoint | Method | Equivalent MCP Tool |
|---|---|---|
| `/api/search` | POST `{"query": "...", "top_k": 5}` | `search_conversations` |
| `/api/conversation/<id>` | GET | `get_conversation` |
| `/api/conversations?offset=0&limit=20&sort_by=date` | GET | `list_conversations` |
| `/api/conversation/<id>/notes/<note_id>` | DELETE | `delete_conversation_note` |
| `/api/stats` | GET | `get_stats` |
| `/api/recluster?k=10` | GET | *(no MCP equivalent)* |

### Claude Code Configuration

Claude Code can also use Constellation as an MCP server. Add to `.claude/settings.json` (project-level) or `~/.claude/settings.json` (global):

```json
{
  "mcpServers": {
    "constellation": {
      "command": "python3",
      "args": ["/absolute/path/to/constellation-v3/server/mcp_server.py"]
    }
  }
}
```

Both Claude Desktop and Claude Code can run simultaneously — each spawns its own subprocess with no port conflicts. The only shared state is the `data/` directory: reads are safe, and note writes use atomic file replacement (`os.replace`) so concurrent writes are safe on POSIX systems.

**Usage from a coding agent**: A coding agent can use `search_conversations` to find past discussions about specific libraries, patterns, or debugging sessions. `list_conversations` lets it browse recent conversations for context without needing a search query. `get_stats` tells it how much memory is available and the date range covered. Notes can be added to conversations to mark important findings or tag conversations for follow-up.

### Data Dependency

The MCP server requires the `data/` directory to be populated by the embedding pipeline. Without it, all tools return errors. The pipeline is triggered by:

```bash
python3 launch.py                    # Full pipeline + serve + browser
python3 launch.py --headless         # Full pipeline + serve (API only)
```

Input data: Place a Claude export JSON file in the project root (the parser looks for `conversations.json` by default, configurable in `config.yaml`).

### Design Decisions (Resolved — V3.2 Scope)

The following questions were evaluated and decided. Items marked **V3.2** are being implemented. Items marked **Deferred** are intentionally out of scope.

| # | Question | Decision | Status |
|---|---|---|---|
| 1 | Notes lost on re-embed | Sidecar `data/notes.json` — notes stored outside regenerated artifacts | **V3.2** |
| 2 | No `list_conversations` tool | Add `list_conversations(offset, limit, sort_by)` | **V3.2** |
| 3 | No `delete_note`/`update_note` | Add `delete_note` only. No update — delete and re-add. | **V3.2** |
| 4 | Cold-start model load | Pre-load embedding model on MCP server startup | **V3.2** |
| 5 | No ingestion via MCP | Defer. Pipeline re-run is the ingestion path. | Deferred |
| 6 | No cluster context in results | Add `cluster_id` + `cluster_label` to search results | **V3.2** |
| 7 | Stats not exposed via MCP | Add `get_stats` tool wrapping existing method | **V3.2** |
| 8 | Message format drops metadata | Include timestamps + notes in `get_conversation` | **V3.2** |
| 9 | Batch operations | Defer. Sequential calls are fine at current scale. | Deferred |
| 10 | No auth on MCP | No action. Stdio is local. Document as future concern for SSE. | Deferred |

### V3.2 Implementation Notes

- The V3.2 spec is maintained externally and given to the coding agent at session start
- Core principle: **do not modify `core/indexer.py`** — the sidecar approach means `save_pipeline_output` no longer needs to care about notes
- New file: `core/notes.py` — standalone note persistence with atomic writes
- Both Claude Desktop and Claude Code can use the same MCP server (each spawns its own subprocess, no conflicts)
- No new dependencies added to `requirements.txt`

## Known Issues and Gotchas

1. **`lexical.py:36` dead variable**: `np_zeros = [0] * self.doc_count` — the name `np_zeros` is assigned but is actually being assigned to `self.doc_lengths` via a chained assignment. Harmless but confusing; `np_zeros` is never referenced again.
2. **`inspector.js:41` dropped `query` parameter**: `renderInspector(node, conv, query)` is called from `showInspector` with 3 args, but the function signature is `renderInspector(node, conv)` — the `query` parameter is silently dropped, breaking query highlighting in the chat viewer.
3. **Palette arrays duplicated**: The DEFAULT_PALETTE and COLORBLIND_PALETTE arrays exist in both `core/indexer.py` (Python) and `frontend/js/graph.js` (JS) with no single source of truth.
4. **No CORS preflight handling**: `http_server.py` adds `Access-Control-Allow-Origin: *` but doesn't handle OPTIONS preflight requests — cross-origin POST to `/api/search` may fail from external clients.
5. **Config merge is shallow-ish**: `config.py:ensure_config()` does a one-level deep merge. Nested config beyond 2 levels won't merge properly.
6. **`parser.py:101` prints to stdout**: `parse_claude_export` uses bare `print()` (not stderr). Safe in the launch pipeline but would corrupt MCP if ever called from MCP context.
7. **HTTP server is synchronous**: `http.server.HTTPServer` is single-threaded; search requests block all other requests (including static file serving) while the embedding model runs.
8. **No request size limits**: `http_server.py:93` reads `Content-Length` bytes with no upper bound check.
9. **XSS in search results**: `search.js:86` injects `n.id` directly into an `onclick` attribute string. If a conversation ID contained a quote character, it could break out of the attribute.
